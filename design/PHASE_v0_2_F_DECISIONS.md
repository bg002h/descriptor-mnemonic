# Phase v0.2-F — Test vector schema bump + dynamic generation decisions

Resolves the open design questions for `8-negative-fixture-dynamic-generation` (and absorbs `phase-d-taproot-corpus-fixtures`) before implementer dispatch.

## F-1 — Generation strategy

**Decision**: generate at vector-build time (Option A from triage). The schema-2 JSON file contains the actually-generated `input_strings` byte-for-byte. No on-the-fly regeneration in test harnesses. The generator function lives in the `vectors` module and is called by `build_test_vectors_v2()`.

**Rationale**: simplest. JSON file IS the authoritative cross-implementation contract. Implementations that want to regenerate locally can do so via the same generator code (open-source); the JSON is the byte-for-byte ground truth. Provenance metadata (which policy + which mutation) can be embedded in the `description` field per fixture.

## F-2 — Schema 2 shape

**Decision**: schema-2 is **additive over schema-1**. Existing fields preserved (`id`, `description`, `policy`, `expected_bytecode_hex`, `expected_chunks`, `expected_wallet_id_words` for `Vector`; `id`, `description`, `input_strings`, `expected_error_variant` for `NegativeVector`). Schema-1 readers can still consume schema-2 files (additive fields ignored).

**New fields in schema-2**:
- `Vector` gains `expected_fingerprints_hex: Option<Vec<String>>` (each entry is 8 hex chars / 4 bytes; lowercase). For non-fingerprints vectors, `None`. For fingerprints vectors, `Some(["deadbeef", "cafebabe", ...])` matching the encoded fingerprints.
- `NegativeVector` gains `provenance: Option<String>` — a one-sentence note about how the `input_strings` were generated (e.g., "encoded `wsh(pk(@0/**))`, then truncated to 5 chars after `wdm1`"). Helps implementers understand WHY a particular byte sequence triggers the variant.

Both fields default to `None` so schema-1 vectors round-trip cleanly into schema-2 form.

**Schema-1 lock unaffected**: `crates/wdm-codec/tests/vectors/v0.1.json` stays at `schema_version: 1`, byte-identical, locked at the existing SHA-256.

## F-3 — File layout

**Decision**:
- `crates/wdm-codec/tests/vectors/v0.1.json` — locked schema-1 file, never modified
- `crates/wdm-codec/tests/vectors/v0.2.json` — NEW schema-2 file, generated by `build_test_vectors_v2()`
- `gen_vectors` extended to support both: `gen_vectors --verify <path>` infers schema from the file's `schema_version` field and dispatches to `build_test_vectors_v1()` or `build_test_vectors_v2()` accordingly

**Backward compat**: existing `gen_vectors --verify v0.1.json` continues to work (calls `build_test_vectors_v1()` internally; schema-1 builder is preserved verbatim from current `build_test_vectors()` body). The new `gen_vectors --verify v0.2.json` calls `build_test_vectors_v2()`.

## F-4 — Negative-fixture regeneration strategy

**Decision**: per-variant generator functions in `vectors.rs`, one per `expected_error_variant`. Each function:

1. Constructs a starting state (a valid policy, a valid bytecode buffer, or hand-crafted bytes)
2. Mutates it precisely to trigger the named variant
3. Encodes to a WDM string (or returns `Vec::new()` if the variant cannot be triggered via a string)
4. Asserts (in a debug build via `debug_assert!`) that decoding the result yields the named variant
5. Returns `(input_strings: Vec<String>, provenance: String)`

**Variants that stay empty** (per the v0.1 honest-provenance docs): `EmptyChunkList` (n12), `PolicyTooLarge` (n30) — these require lower-level API calls, not WDM strings. Provenance documents which API.

**Variants that get real strings now** (was placeholder): `MixedCase`, `InvalidStringLength`, `InvalidChar`, `BchUncorrectable`, `UnsupportedVersion`, `UnsupportedCardType`, etc. — most of v0.1's `n02..n28` set.

**New variants from v0.2 work**: `TapLeafSubsetViolation` (Phase D), `FingerprintsCountMismatch` (Phase E). Add as new negative entries.

## F-5 — Generator stability

**Decision**: the `build_test_vectors_v2()` function is **deterministic** — same code → same output. No RNG, no system-time, no env-var dependence. The `generator: String` field embeds the wdm-codec version (`"wdm-codec v0.2.0-dev"`) so cross-version regeneration is detectable.

## F-6 — SHA-256 lock for v0.2.json

**Decision**: after generating `v0.2.json`, compute its SHA-256 and embed in:
- `crates/wdm-codec/tests/vectors_schema.rs` — pinning test asserts the committed file's SHA matches the expected hash (defense against accidental edits)
- `bip/bip-wallet-descriptor-mnemonic.mediawiki` §"Test Vectors" — replace the v0.1 SHA reference with the v0.2 SHA, alongside an addendum noting v0.1.json is also still authoritative for the v0.1.0 release

## F-7 — Taproot corpus additions (absorbed `phase-d-taproot-corpus-fixtures`)

**Decision**: add 3 positive corpus entries to `CORPUS_FIXTURES` for v0.2:
- `tr_keypath` — `tr(@0/**)` (key-path-only, no script tree)
- `tr_pk` — `tr(@0/**, pk(@1/**))` (single-leaf pk)
- `tr_multia_2of3` — `tr(@0/**, multi_a(2,@0/**,@1/**,@2/**))` (single-leaf multi_a)

These appear in `build_test_vectors_v2()` but NOT in `build_test_vectors_v1()` (so v0.1.json stays byte-identical).

Add 1-2 negative entries:
- `n_tap_leaf_subset` — `tr(@0/**, sha256(...))` → `TapLeafSubsetViolation { operator: "sha256" }` (with real string from encoder + decoder error)
- `n_taptree_multi_leaf` — hand-crafted bytecode with `Tag::TapTree=0x08` in leaf position → `PolicyScopeViolation` with multi-leaf message

## F-8 — Fingerprints corpus additions

**Decision**: add 1 positive corpus entry to `CORPUS_FIXTURES` for v0.2:
- `multi_2of2_with_fingerprints` — `wsh(multi(2,@0/**,@1/**))` encoded with `EncodeOptions::default().with_fingerprints(vec![Fingerprint::from([0xde,0xad,0xbe,0xef]), Fingerprint::from([0xca,0xfe,0xba,0xbe])])`. Tests the schema-2 `expected_fingerprints_hex` field.

Add 1-2 negative entries:
- `n_fingerprints_count_mismatch` — encoder rejection (fps.len() != placeholder count) — `FingerprintsCountMismatch { expected, got }`
- `n_fingerprints_missing_tag` — hand-crafted bytecode with header bit 2 set but no `Tag::Fingerprints` after path → `InvalidBytecode { kind: UnexpectedTag { expected: 0x35, got: <other> } }`

## F-9 — `gen_vectors` CLI changes

**Decision**: minimal additive change. The existing `--verify <path>` and `--output <path>` flags stay. The verify path now reads the file's `schema_version` field first and dispatches to the correct builder. The output path needs a new `--schema <1|2>` arg (default 2 for the new v0.2.json) so callers can regenerate v0.1.json verbatim if needed (rare; mostly for spec recovery).

```
gen_vectors --output <path>            # writes schema-2 by default
gen_vectors --output <path> --schema 1  # writes schema-1 (for v0.1.json regen)
gen_vectors --verify <path>            # infers schema from file
```

## F-11 — Test-harness changes

**Decision**: `crates/wdm-codec/tests/vectors_schema.rs` gets new tests:
- `committed_v0_2_json_matches_regenerated_if_present` — schema-2 mirror of the existing v0.1 test
- `v0_2_sha256_lock_matches_committed_file` — SHA-256 lock test (defense against accidental edits)

The existing `committed_json_matches_regenerated_if_present` test stays and continues to verify v0.1.json byte-identical.

## F-12 — How does the encoder for schema-2 fingerprints vector get the fingerprints?

**Decision**: the schema-2 `Vector` gets a new `encode_options_fingerprints: Option<Vec<[u8; 4]>>` field (sibling to the existing `policy: String`). When the vector is constructed via the generator, the field carries the fingerprints used during encoding. The generator passes them to `EncodeOptions::default().with_fingerprints(...)` to produce `expected_bytecode_hex`, `expected_chunks`, `expected_fingerprints_hex`.

For non-fingerprints vectors, `encode_options_fingerprints == None` and the encoder uses `EncodeOptions::default()`.

(Equivalent decision could be made for `EncodeOptions::shared_path` if any vector wants a non-default path, but for v0.2 we don't add path-override vectors.)
