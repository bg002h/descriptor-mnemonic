## MEDIUM

### M-1: Test helpers panic silently on encode/decode failure

`encode()` and `decode()` helpers in `template_roundtrip.rs` and `json_snapshots.rs` chain
`stdout.lines().next().unwrap()`. If the CLI exits non-zero, stdout is empty and the panic message
gives no clue which manifest entry failed or what stderr said. Plan didn't require status check;
flagging for a follow-up:

```rust
fn encode(template: &str) -> String {
    let out = Command::cargo_bin("md").unwrap().args(["encode", template]).output().unwrap();
    assert!(out.status.success(), "md encode {:?} failed: {}", template,
        String::from_utf8_lossy(&out.stderr));
    String::from_utf8(out.stdout).unwrap().trim().to_string()
}
```

### M-2: `wsh_with_fingerprints` snapshot doesn't actually use fingerprints

The `encode()` helper passes only `v.template`, ignoring `v.keys` / `v.fingerprints`. The
`wsh_with_fingerprints` snapshot has `"fingerprints": null`. Coverage for the fingerprint TLV
path is preserved transitively by `vector_corpus.rs` (which uses `cmd::vectors::run` that does
pass fingerprints). Plan didn't require explicit fingerprint flag plumbing here.
