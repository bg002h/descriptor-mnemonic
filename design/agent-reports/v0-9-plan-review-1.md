# v0.9.0 plan review (opus, pass 1)

**Date:** 2026-04-29
**Plan:** design/IMPLEMENTATION_PLAN_v0_9_chunk_set_id.md
**Reviewer:** opus-4.7

## Summary

**Verdict: address-then-proceed.** The three-phase shape is correct, the wire-additivity argument for `0x16` is sound, and the cross-repo stewardship piece is well-scoped. However the rename map is materially incomplete in one large-blast-radius way: the v0.8 rename `WalletId → PolicyId` only updated the *type spelling* of the chunk-header field, not the *variant names* of two chunk-level errors that travel with it. `Error::PolicyIdMismatch` and `Error::ReservedPolicyIdBitsSet` are CHUNK-level errors carrying `ChunkPolicyId` payloads and prose about "wallet-id bits" — they should rename in lockstep with `ChunkPolicyId → ChunkSetId` (becoming `Error::ChunkSetIdMismatch` and `Error::ReservedChunkSetIdBitsSet`). The plan currently asserts these must NOT change, which would leave the surface inconsistent: a `ChunkSetId` would still throw `Error::PolicyIdMismatch`. Three concrete fixes below; once those are in, the plan is green.

## Findings

### F1: `Error::PolicyIdMismatch` and `Error::ReservedPolicyIdBitsSet` are misclassified as Tier-3 — they are chunk-level

**Severity:** blocker
**Location:** plan §"Naming map (P1)" line 39 (out-of-rename list); plan §"Phase 1 step 1"; cross-checked against `crates/md-codec/src/error.rs:152-164,230-236` and `crates/md-codec/src/chunking.rs:683`

**Issue:** The plan's out-of-rename list includes `Error::PolicyIdMismatch` under the rationale "Tier-3 PolicyId mismatch — confirm in P1 step 1." But the variant's payload is `expected: ChunkPolicyId, got: ChunkPolicyId` and it fires from `reassemble_chunks` when two chunks disagree about the chunk-header 20-bit field (`chunking.rs:682-685`). It is the variant the plan's *own* P1 step 1 audit grep is trying to find — the historical name is misleading because the v0.8 rename `WalletId → PolicyId` mechanically renamed `WalletIdMismatch → PolicyIdMismatch` without recognising that this error is in the chunk-id domain, not the Tier-3 PolicyId domain.

`Error::ReservedPolicyIdBitsSet` has the same problem — its rustdoc literally reads "The three wallet-id bytes in a chunk header had the reserved top 4 bits set" (`error.rs:230-236`). It is a chunk-header parser error, not a Tier-3 PolicyId error. (And the Tier-3 16-byte PolicyId has no "reserved bits" concept anyway — the SHA-256 prefix uses all 128 bits.)

If both stay, the v0.9 surface is inconsistent: `ChunkSetId::new(0x10_0000)` would panic constructively, but a malformed chunk header with the same bit pattern would surface as `Error::ReservedPolicyIdBitsSet`. Two `ChunkSetId` values disagreeing across chunks would surface as `Error::PolicyIdMismatch`. A user reading the rustdoc would be left wondering which "policy-id" is meant.

**Recommendation:**
1. Add to the naming map:
   - `Error::PolicyIdMismatch { expected, got }` → `Error::ChunkSetIdMismatch { expected, got }`
   - `Error::ReservedPolicyIdBitsSet` → `Error::ReservedChunkSetIdBitsSet`
   - The error message strings ("wallet identifier mismatch across chunks: …", "reserved wallet-id bits set: …") rewrite to use "chunk-set identifier" / "chunk-set-id".
2. Update P1 step 1 to *expect* and *prescribe* this rename rather than treat it as a gating audit question.
3. The 18 sites that reference these variants by name (rg shows hits in `error.rs`, `chunking.rs`, `decode.rs`, `vectors.rs:557,608,613,1343,2269,2275`, `tests/conformance.rs:192,197,295,318`, `tests/error_coverage.rs:48,56`, both `v0.1.json` and `v0.2.json` corpus files via `expected_error_variant` keys) must rename in lockstep. The v0.1.json corpus is not regenerated by `gen_vectors --schema 1` byte-identically when error variant strings change — so this is a wire-format-prose change for the JSON corpus and the SHA pin update logic in P4 step 4 needs to apply to BOTH v0.1.json AND v0.2.json.

This is the difference between "pure docs+symbols rename, no wire-format change" (still true) and "test-corpus JSON content changes" (also true, and not currently called out).

### F2: ChunkHeader struct field `policy_id: ChunkPolicyId` and ~30 chunking.rs prose / variable mentions need rename

**Severity:** strong
**Location:** plan §"Phase 1 step 2" sed sweep; cross-checked against `crates/md-codec/src/chunking.rs:67,87,91,152,167,537,589,661,664,677,682,685,763,776,795,830,845,1217,1247,1474,1495,1504,1525,1529`

**Issue:** The sed pattern `s/\bchunk_policy_id\b/chunk_set_id/g` does not match the field name `policy_id:` on `ChunkHeader::Chunked` (the field literally lives as `policy_id`, not `chunk_policy_id`). Numerous local variables also use the bare name `policy_id` in match-binding context (`ChunkHeader::Chunked { policy_id, count, .. }` destructuring; `expected_wallet_id`, `test_wallet_id()`, `wid_a/wid_b`, `chunked_round_trip_max_wallet_id`).

Running the proposed sed unchanged will:
- Leave the `policy_id` field on `ChunkHeader::Chunked` unrenamed (semantic confusion: the field is a `ChunkSetId` but spelled `policy_id`).
- Leave `expected_wallet_id`, `test_wallet_id()`, `chunked_round_trip_max_wallet_id` un-renamed.
- Leave the prose comments "wallet identifier", "wallet-id", "wallet-id bits" etc. (~21 occurrences in chunking.rs alone) unrenamed.

The bare name `policy_id` is also used pervasively as a sed-anchor risk: a global `s/\bpolicy_id\b/chunk_set_id/g` would FALSELY hit `compute_policy_id`, `policy_id_words`, `expected_policy_id_words`, `compute_policy_id_for_policy`, and `policy_id_seed` (the very thing we want to rename to `chunk_set_id_seed` but only when the prefix is right). So a blanket bare-`policy_id` rewrite is too dangerous; the rewrite must be context-aware.

**Recommendation:**
- Add an explicit step: rename the `policy_id:` field on `ChunkHeader::Chunked` (and `Chunk`) to `chunk_set_id:`. This is a structural code edit; not a sed candidate. It will cascade into ~25 destructuring sites in `chunking.rs`, `vectors.rs`, `tests/`.
- Add an explicit sweep for `wallet_id` / `wallet-id` / "wallet identifier" / "wallet identif" in code comments AND test-helper names (`test_wallet_id`, `chunked_round_trip_max_wallet_id`, `expected_wallet_id`, `wid` / `wid_a` / `wid_b`). Suggested replacements: `test_chunk_set_id`, `chunked_round_trip_max_chunk_set_id`, `expected_chunk_set_id`, `csid` / `csid_a` / `csid_b`.
- The grep `rg -nc 'wallet[-_ ][Ii]dentif|wallet[-_ ]ID|wallet[-_ ]id\b' crates/md-codec/ design/POLICY_BACKUP.md MIGRATION.md README.md` returns **76 hits** — well beyond what the plan's per-file count map estimates. Update plan's per-file ref-count table accordingly so reviewers know the true blast radius.

### F3: `s/policy_id_seed/chunk_set_id_seed/g` — false-positive surface is small but real

**Severity:** nice-to-have
**Location:** plan §"Phase 1 step 2"

**Issue:** The user's review prompt asked specifically whether the sed would accidentally hit `policy_id_seed_documentation` or other compounds. Cross-checked: `rg 'policy_id_seed[a-zA-Z_0-9]' crates/md-codec/` returns ZERO compound hits — the only suffix observed is the bare term, the field, and prose with backticks. So the proposed sed is safe in *this* dimension.

Adjacent risk: `s/policy_id_seed\([_a-zA-Z0-9]\)/chunk_set_id_seed\1/g` in step 2 (the second sed in the snake-case block) is unnecessary and slightly wrong: it would rewrite `policy_id_seedX` → `chunk_set_id_seedX`, but no such symbol exists, and the regex risks matching across a multi-line trailing alphanumeric in a corner case that doesn't apply here. The `\b…\b` form in the prior line is sufficient.

**Recommendation:** Delete the two compound-suffix sed patterns (`s/chunk_policy_id\([_a-zA-Z0-9]\)/…` and `s/policy_id_seed\([_a-zA-Z0-9]\)/…`) in step 2; the `\b` word-boundary form covers the whole surface, and the compound-suffix forms add risk without adding coverage.

(False alarm worth confirming: `compute_policy_id_for_policy`, `compute_policy_id`, `policy_id_words`, `expected_policy_id_words`, `PolicyIdMismatch`, `PolicyIdWords`, `PolicyIdSeed`, `ReservedPolicyIdBitsSet` — none of these contain the literal substring `policy_id_seed` or `chunk_policy_id`. All are safe under the `\b...\b`-only approach.)

### F4: 0x16 wire-additivity claim is correct; one concrete corpus negative-vector pin needs to be DELETED (not just stop expecting rejection)

**Severity:** strong
**Location:** plan §"Phase 2 step 6"; cross-checked against `crates/md-codec/src/bytecode/path.rs:363,512`

**Issue:** Two test arrays in `path.rs` explicitly enumerate `0x16` as a rejected indicator:

- Line 363, `decode_rejects_reserved_indicator`: `for &b in &[0x00u8, 0x08, 0x10, 0x16, 0x18, 0xFD, 0xFF]` — pins `decode_path` rejecting `0x16` with `UnknownTag(0x16)`.
- Line 512, `unknown_indicator_returns_none`: `for &ind in &[0x00u8, 0x08, 0x10, 0x16, 0x18, 0xFD]` — pins `indicator_to_path(0x16)` returning `None`.

After v0.9 ships, both tests must DROP `0x16` from these arrays (the plan's step 6 calls this out for line 363 but says "remove 0x16 from the 'rejected indicators' arrays" — please read this in the singular: "arrays" is plural and both line 363 and line 512 are involved; double-check the implementer follows through).

I did NOT find any byte-literal corpus negative-vector in `tests/vectors/v0.1.json` or `tests/vectors/v0.2.json` that pins a chunked-string with `0x16` payload as a rejected vector. So at the corpus level this is a clean wire-additive change. Confirmed.

The plan's negative-test surface adjustment is incomplete in one detail: the rustdoc on `decode_path` at line 89 includes the literal text "(0x01–0x07, 0x11–0x15, 0x17)" — the plan's step 5 catches this. ✓

**Recommendation:** In Phase 2 step 6, list both line 363 AND line 512 explicitly (the current "lines ~363 and ~512" formulation is ambiguous about which test does what). Also: the plan's step 6 says "Update the positive table at ~484 to include `(0x16, "m/48'/1'/0'/1'")`." Confirmed that `FIXTURE` lives at `path.rs:479-493`; step 6's instruction is correct in substance but the implementer should be careful to insert the new tuple in sorted order (between `0x15` and `0x17`) so the FIXTURE matches the BIP table layout.

### F5: BIP-submission precondition framing — recommend coordinated draft-PR-first

**Severity:** strong
**Location:** plan §"Phase 4 step 10"

**Issue:** The plan ships md1 v0.9.0 first, then opens a sibling PR in `bg002h/mnemonic-key`. But the FOLLOWUPS entry says "shipping this rename is a hard precondition for mk1's BIP submission" (FOLLOWUPS.md:93) — meaning mk1 cannot land its BIP submission until md1 v0.9.0 ships. If the md1 PR uncovers a *different* naming choice during review (e.g., `chunk_set_id` vs `chunk_id` vs `csid`), mk1 has to track through.

A safer dance:
- Open mk1 draft PR in parallel using the proposed `chunk_set_id` terminology.
- Cross-link: md1's release-PR body links the mk1 draft; mk1 draft's body links md1's release-PR.
- Land md1's PR → tag md1-v0.9.0 → flip mk1 draft to ready-for-review with the resolved md1 commit cited.

This way reviewers see the full cross-repo shape from day one and any naming pushback on md1's PR catches mk1 too.

**Recommendation:** Restructure Phase 4 step 10 to a Phase 0 step (open coordinated draft PR in mk1 *before* md1 PR, both citing each other) and a Phase 4 step 11 (after md1 ships, flip mk1 draft to ready-for-review with the md1 commit pinned). This is also consistent with the new RELEASE_PROCESS.md doc the plan creates in P3, which explicitly says "Open a coordinated PR in `bg002h/mnemonic-key` updating mk1's spec to mirror md1's table" — that doc's own checklist says "before tagging the md1 release."

### F6: Hot-on-heels rename — CHANGELOG/MIGRATION should acknowledge prominently

**Severity:** nice-to-have
**Location:** plan §"Phase 4 step 2"

**Issue:** v0.8.0 (2026-04-28) just renamed `WalletId → PolicyId` across ~720 references and ~40 files; v0.9.0 (proposed: 2026-04-29, one day later) renames `ChunkPolicyId → ChunkSetId` across a sub-domain of that just-renamed surface. Reviewers and downstream callers will reasonably ask "wait, is this name going to change again?"

Suggest concrete prose for the CHANGELOG `[0.9.0]` opener:

> ### Why a rename, *again*?
>
> v0.8.0 renamed `WalletId → PolicyId` to align with BIP 388's policy-template
> framing (Tier-3 = "the policy"). That rename mechanically renamed the
> chunk-header 20-bit field `ChunkWalletId → ChunkPolicyId` along with it —
> but on review for the mk1 BIP submission, the chunk-header field is neither
> a Policy ID nor a Wallet Instance ID; it identifies the chunk-set assembly
> and nothing more. v0.9.0 corrects the chunk-header field's name to
> `ChunkSetId`. v0.8's `PolicyId` and `WalletInstanceId` are stable; this
> rename touches only the chunk-header sub-domain (~150 references, single
> module). We expect this to be the last identifier rename in this family.

**Recommendation:** Add a similar "Why a rename, again?" callout to the top of the new CHANGELOG entry AND a parallel paragraph at the top of the MIGRATION v0.8.x → v0.9.0 section. This frames the rename as "correcting collateral damage from v0.8" rather than "we change names every week," which is a much better story for the BIP submission.

### F7: Phase 2 vectors regen — SHA pin sequencing across schema-1 vs schema-2

**Severity:** nice-to-have
**Location:** plan §"Phase 2 step 8" + §"Phase 4 step 4"

**Issue:** The plan regenerates `tests/vectors/v0.2.json` in P2 step 8 and updates the SHA pin in P4 step 4. The `gen_vectors` binary supports BOTH schema-1 (regenerates `tests/vectors/v0.1.json`) and schema-2 (regenerates `v0.2.json`); v0.1.json is "byte-identical regen" historically. With F1 applied (error-variant rename), `expected_error_variant` strings in v0.1.json would change, breaking the byte-identical regen invariant.

If F1 is accepted, P4 step 4 must regen BOTH schema-1 and schema-2 corpora and update both SHA pins. If F1 is rejected, v0.1.json is byte-identical and only v0.2.json's SHA pin needs updating.

**Recommendation:** Add explicit sub-steps to P4 step 4:
```bash
cargo run --features test-helpers --bin gen_vectors -- --output crates/md-codec/tests/vectors/v0.1.json --schema 1
cargo run --features test-helpers --bin gen_vectors -- --output crates/md-codec/tests/vectors/v0.2.json
sha256sum crates/md-codec/tests/vectors/v0.1.json crates/md-codec/tests/vectors/v0.2.json
```
…and update both SHA pins in `tests/vectors_schema.rs`.

### F8: 0x16 BIP cross-reference — Tag::OrB has the same byte value (different namespace, but worth a comment)

**Severity:** nice-to-have
**Location:** plan §"Phase 2 step 9"; cross-checked against `crates/md-codec/src/bytecode/tag.rs:80,165,242` and `bip/bip-mnemonic-descriptor.mediawiki:420`

**Issue:** `Tag::OrB = 0x16` is also defined in the operator-tag namespace. The path-dictionary indicator `0x16` and the operator tag `Tag::OrB` both exist at `0x16`. They cannot collide (different parser positions) but a reader scanning the BIP for `0x16` will see TWO definitions and may pause.

**Recommendation:** In Phase 2 step 9, the new BIP table entry should optionally have a sub-note: `(distinct from operator tag <code>0x16</code> "or_b" — different parsing context).` Or simply do nothing; the BIP's table headers already make the namespace context clear. Calling it out is paranoia-grade documentation.

## Confirmations

- The naming map's core entries (`ChunkPolicyId → ChunkSetId`, `PolicyIdSeed → ChunkSetIdSeed`, `EncodeOptions::policy_id_seed → chunk_set_id_seed`) are correct and complete in their direct surface.
- The out-of-rename guardrails for `PolicyId`, `PolicyIdWords`, `WalletInstanceId`, `compute_policy_id_for_policy`, `compute_wallet_instance_id` are correct: those symbols correctly identify the Tier-3 / template-instance domains and have no chunk-id semantics.
- The plan's per-file count map (`policy_id.rs ~58`, `chunking.rs ~20`, `options.rs ~17`, etc.) matches `rg -nc` output ±1.
- The wire-additivity argument for `0x16` is correct: existing decoders return `UnknownTag(0x16)` for `0x16` payloads (`path.rs:155-162`), and there is no byte-literal corpus negative-vector pinning `0x16` as rejected.
- The TDD-style P2 step 2 (write failing test first) is appropriate and matches the repo's executing-plans skill.
- The choice to keep `md-per-at-N-path-tag-allocation` deferred (not in v0.9 scope) is correct — wire-format-breaking changes belong in their own release.
- `Error::WalletIdMismatch` does NOT exist (rg returned zero hits); the audit step in P1 step 1 will confirm this. The actual problematic variant is `Error::PolicyIdMismatch` per F1.
- The proposed sed's `\bpolicy_id_seed\b` form does NOT false-positive on `policy_id_words`, `policy_id_for_policy`, `compute_policy_id`, etc. (verified by rg).
- The `s/\bChunkPolicyId\b/ChunkSetId/g` is safe: there is no compound CamelCase form `ChunkPolicyIdSomething` in the codebase.
- Phase 3 (RELEASE_PROCESS.md) is well-scoped and correct prose.
