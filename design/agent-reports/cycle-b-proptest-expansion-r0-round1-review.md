## Critical (must fix before any further step)

- **[C1] P6's reparse oracle spuriously fails for every tr() descriptor whose tap leaf is well-typed but not "sane" — the grammar must guarantee per-leaf SANITY in the Tap context, not just type-correctness.** In pinned miniscript-13.0.0, `Miniscript::from_ast` (miniscript/mod.rs:298) runs only `Type::type_check` + `check_global_validity` — it does NOT reject sigless/malleable expressions (proven: `from_ast(or_i(older(1),older(2)))` = Ok, `sanity_check()` = `Err(SiglessBranch)`). `Descriptor::new_tr` (descriptor/mod.rs:246 → tr/mod.rs:96) does no leaf sanity either. But `Descriptor::from_str` (descriptor/mod.rs:1047–1062) carries a **Tr-only** 12.x-compat branch: `ret.sanity_check()` + per-leaf `ext_check(ExtParams::sane())`. Proven empirically: `tr(xpub…/0/*,older(144))` constructs via `new_tr`, renders, and **reparse fails with "All spend paths must require a signature"**; `tr(…, or_i(pk(A),pk(A)))` reparse fails with "repeated pubkeys". The spec's Tap production pool (bare hashlocks, bare timelocks, `or_i(B,B)` with arbitrary B, no per-leaf key-distinctness, no height/time-mixing exclusion — spec :131–138) generates non-sane tap leaves wholesale → a large fraction of tr cases fail P6's "MUST reparse" leg through no codec bug. wsh/sh are NOT affected (proven: `wsh(or_i(older(1),older(2)))` reparses Ok — `Wsh::from_tree` does only `top_level_checks`, segwitv0.rs:251–259). Fix: Tap-context T productions must additionally guarantee `sanity_check` per leaf — every spend path signature-bearing (e.g. leaves shaped `and_v(v:<lock/hash>, <sig-bearing B>)`, `pk`, `multi_a`, sig-bearing combinators), distinct `@i` within a leaf, no height/time relative- or absolute-lock mixing within a leaf — OR weaken the tr reparse oracle (not recommended; sanity-by-construction mirrors Cycle A and is what real wallets need anyway).

- **[C2] The spec's locktime-rejection claim is factually wrong, and a P7 oracle self-test fixture is built on it.** Spec :71–74 claims `RelLockTime::from_consensus` rejects "older() values with bits outside the BIP-68 mask 0x0040FFFF", and :166 pins `older(0x00410000)` as a P7 known-bad fixture. Measured against pinned miniscript-13.0.0 (primitives/relative_locktime.rs:42–74, `TryFrom<Sequence>`: accepts iff `seq.is_relative_lock_time() && seq != ZERO`, i.e. **only bit 31 and zero are checked**): `older(0x10000)` Ok, `older(0x0040FFFF)` Ok, **`older(0x00410000)` Ok**, `older(0x7FFFFFFF)` Ok, `older(0x80000000)` Err, `older(u32::MAX)` Err, `older(0)` Err. (This is exactly why the toolkit needed its own v0.53.9 BIP-68-mask gate — md-codec's converter inherits miniscript's leniency.) The P7 fixture as written asserts `Err` and gets `Ok` → permanent red, and the same false premise contaminates the T-tier "consensus-valid domains" and the W-tier P7 input classification. Fix: rel-bad = {0} ∪ {bit-31 set}; rel-good includes the out-of-mask values 0x10000/0x00410000 (P6-constructible, and arguably worth pinning Ok loudly with a comment citing toolkit v0.53.9). Abs domain confirmed as spec'd: `AbsLockTime::from_consensus` valid 1..=0x7FFF_FFFF (absolute_locktime.rs:10,17,51); after(0)/after(0x80000000)/after(u32::MAX) Err — rejection happens at the constructor call inside `node_to_miniscript` (to_miniscript.rs:429–435), a clean Err, never `from_ast`.

- **[C3] Keyless W-tier trees encode-fail: n = 0 is rejected at `PathDecl::write`, a spurious-failure trap the spec never mentions.** The W leaf pool includes `True`, `False`, timelocks, and hashlocks (spec :100–104), and the combinator layer can compose trees with **zero** placeholder references (e.g. `wsh(older(5))`). `descriptor_from_tree` then derives n = 0, and `encode_payload` → `PathDecl::write` returns `Err(KeyCountOutOfRange { n: 0 })` (origin_path.rs:105–111; proven empirically). P1/P2/P4/P5 all `.expect("encodes")` → spurious failure. Fix: the W strategy must guarantee ≥ 1 key-bearing node by construction (e.g. always conjoin one `PkK` leaf, or restrict at least one branch of the top combinator to the key-bearing sub-pool). Same applies to any taptree whose leaves are all keyless when `is_nums = true`.

## Important (must fix before implementation)

- **[I1] The recursive `or_d(...)` member of the spec's Bdu set is not Bdu in general.** `or_d(X,Z)` propagates `d`/`u` from its **right** arm; spec :134 puts `or_d(...)` in the Bdu pool while defining the production as `or_d(Bdu, B)`. Proven: `or_d(or_d(pk,older),pk)` fails `from_ast` ("requires its left child be dissatisfiable"); `or_d(or_d(pk,pk),pk)` is Ok. Fix the recursion: a Bdu-position `or_d` must be `or_d(Bdu, Bdu)` (right arm also du). The rest of the typing claims check out empirically: `thresh(k, Bdu, W…)` correct (bare B in rest-position rejected: "sub-fragment 1 has type B rather than W"; both `s:` and `a:` over du-leaves accepted); timelocks correctly excluded from Bdu (`or_d(older,pk)`, `andor(older,pk,pk)`, `thresh(1,older,s:pk)` all rejected, the last because older is not a unit); hashlocks ARE valid Bdu (`or_d(sha256,pk)` Ok); `and_v(v:B,B)`, `or_i(B,B)`, `and_b(B,W)` sound.

- **[I2] Per-context resource caps missing from the T grammar — `from_ast` enforces them and will spuriously fail P6.** Verified in pinned source + experiment: (a) **Legacy**: `pk_cost ≤ 520` (`MAX_SCRIPT_ELEMENT_SIZE`, context.rs:381–410) — `sh(multi(2, n=16))` rejected at 547 bytes, n=15 Ok; general `sh(<miniscript>)` hits 520 after ~14 keys' worth of script, so Legacy nesting depth must be tightly capped. (b) **Segwitv0**: `Threshold<_, MAX_PUBKEYS_PER_MULTISIG=20>` — `multi` with n ∈ 21..=32 fails in `build_multi_threshold`; plus `pk_cost ≤ 3600` policy cap (context.rs:531). (c) **Tap**: `multi_a` fine to 32 (`MAX_PUBKEYS_IN_CHECKSIGADD` = 999). The wire allows k ≤ n ≤ 32 everywhere, so the T grammar needs per-context n caps (Legacy multi ≤ 15 and shallow; Segwitv0 multi ≤ 20) and a context-aware size budget. The 21..=32 multi shapes become P7 inputs (wire-valid, loud refuse).

- **[I3] 64-chunk cap headroom is razor-thin for xpub-attached descriptors — state an explicit payload budget.** `split()` hard-errors at > 64 chunks, i.e. payload > 20,480 bits (chunk.rs:251). Measured: n=32 + 32 pubkeys + 32 fingerprints + 32 divergent 3-deep paths + a minimal wsh(multi) tree = **19,855 bits = 63 chunks** — ~600 bits (≈ 2 hash leaves) below the cap. Any T-tier case near n=32, or any W-tier tree rich in 262-bit hash leaves (cap ≈ 78 Sha256 leaves even with empty TLV), exceeds it → `Err(ChunkCountExceedsMax)` → P5/P6 spurious failure. Fix: cap T-tier n (e.g. ≤ 16 when attaching full TLV) and give the W tier an explicit `prop_recursive` node budget sized against the 2,560-byte payload ceiling; document the arithmetic in the spec. (Observation, non-blocking: chunks already exceed the 80-char regular form — 88 chars at the 40-byte/chunk sizing — but stay ≤ 93, within BCH design length; pre-existing, not introduced by this cycle.)

- **[I4] P6's middle oracle is vacuous: "decoded's rendering == original's rendering" is implied by P1/P4/P5 + determinism.** P1/P4/P5 already assert decoded AST == canon(original); `to_miniscript_descriptor` is a pure function of the AST, so string equality of the two renderings can never fail independently. What P6 genuinely adds is (a) `to_miniscript` SUCCEEDS on the whole typed domain and (b) the rendered string is a real descriptor. Strengthen (b): assert **reparse fixed-point as semantic equality** — `Descriptor::from_str(rendered) == constructed Descriptor` (rust-miniscript `Descriptor` is `PartialEq`), which catches Display/parse asymmetries that "parses at all" misses; optionally add the Cycle-A-style differential — `at_derivation_index(0)` → `address()` on the reparsed descriptor vs md-codec's own `derive_address`, which would be the only oracle here not derived from md-codec's converter itself.

- **[I5] "derive up to 33 account xpubs" (spec :127) exceeds the wire's n ≤ 32.** `PathDecl::write` rejects n outside 1..=32 (origin_path.rs:111), and `Body::MultiKeys`/`Variable` are capped at 32. A tr with a non-NUMS internal `@0` plus a 32-key `multi_a` would need n = 33 → encode fails. Cap key material at 32 (and per I3, the practical T-tier cap is lower anyway).

## Answers to open questions

1. **Typed grammar vs full type system**: typed grammar is the right call (Cycle-A precedent). The production list is sound after three repairs: (i) recursive Bdu `or_d` must be `or_d(Bdu, Bdu)` [I1]; (ii) Tap productions must guarantee per-leaf *sanity*, not just types [C1]; (iii) per-context size/key caps [I2]. `thresh(k, Bdu, W…)` and `and_b(B, W)` verified sound exactly as written (first thresh arg Bdu, rest W via `s:`/`a:` over du-units).
2. **Strings vs Descriptor equality**: neither as proposed — the original-vs-decoded comparison is vacuous [I4]. The load-bearing assertion is constructed-vs-REPARSED `Descriptor` equality (semantic fixed-point), with the address differential as the strongest optional add.
3. **File layout**: agree with the proposal — strategies in `tests/common/mod.rs`, W properties join `proptest_roundtrip.rs`, P6/P7 in a new `proptest_to_miniscript.rs` (it needs `#![cfg(feature = "derive")]` like address_derivation.rs).
4. **Budget**: 256 cases is fine. `OnceLock` xpubs: yes — deriving ≤ 32 account xpubs once is milliseconds; per-case `to_miniscript` + reparse is tens of µs. The slow leg is W-tier P5 (multi-chunk BCH over hash-heavy trees) — the I3 node budget also keeps wall-clock sane.
5. **Encode never-panic property**: yes, cheap and worth it. Two specifics: (a) k > n with both ≤ 32 *encodes successfully* but decode rejects `KGreaterThanN` — a real encoder-side gap (an engrave-but-can't-restore shape at the codec level, same family as the Cycle-A find); pin it loudly and file a FOLLOWUP rather than hide it in the generator constraint. (b) `write_bits` masks rather than panics, so encode-side panics are unlikely; the property is mostly about clean `Err` on the 1..=32 range gates.
6. **Origin paths**: randomize shared/divergent (+ occasional `origin_path_overrides`/`fingerprints`/`pubkeys` even in the W tier) — it exercises `canonicalize_placeholder_indices`' TLV-permutation arm (canonicalize.rs:221–232), which currently has zero property coverage. Constraints: paths must be non-empty (the existing helper's Shared([84']) already satisfies `validate_explicit_origin_required` for non-canonical wrappers), `use_site_path_overrides` must keep the alt-count uniform (validate.rs:117–138), and total TLV bits count against the I3 budget.
