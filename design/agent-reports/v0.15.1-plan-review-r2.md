## New-issue checks on the M1 wsh-multi test

### Q1 — Duplicate key at `parse_template` / `multi(...)`?

No issue. `parse_template` uses **synthetic keys** (one per distinct `@i`
index); it never sees the real xpubs. `synthetic_xpub_for(0, MultiSig)` !=
`synthetic_xpub_for(1, MultiSig)` (unit test at template.rs:298-301 proves
this). Miniscript parses the substituted template with two distinct
synthetic keys and does not reject it. The plan-comment at the test (lines
1312-1315) is accurate.

### Q2 — `bitcoin::PublicKey::new(cpk.0)` API correctness?

Correct. `CompressedPublicKey.0` is `secp256k1::PublicKey`;
`bitcoin::PublicKey::new` accepts that. The library itself uses
`bitcoin::PublicKey::from(CompressedPublicKey)` (derive.rs:207) — a
different but equivalent route; both produce a compressed
`bitcoin::PublicKey` with the same 33-byte serialization. The cross-check
in the test will match the library's derivation byte-for-byte.

### Q3 — Builder chain for wsh-multi script?

Confirmed valid in bitcoin 0.32.0. `push_int(i64)`, `push_key(&PublicKey)`,
`push_int(i64)`, `push_opcode(Opcode)`, `into_script()` all present with
those exact signatures. `into_script()` returns `ScriptBuf`, which derefs
to `&Script` for `Address::p2wsh` — no type mismatch.
