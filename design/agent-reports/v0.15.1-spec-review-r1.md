### H1 â€” `AddressArgs` lifetime param dispatch pattern not specified

Â§Implementation surface defines `AddressArgs<'a>` with `&'a [String]` /

### H2 â€” `encode --json` `network` field: serialization vocabulary not specified

The SPEC explicitly calls this out for `address --json` ("mirrors the CLI
flag vocabulary, NOT `bitcoin::Network`'s `Display`") but does not repeat the
constraint for the `encode --json` `network` field. An implementer that
serializes `args.network.to_string()` will write `"bitcoin"` â€” silently
breaking the JSON consumer contract.

**Remediation:** explicitly state that the `encode --json` `network` field
uses the `CliNetwork` ValueEnum string (kebab-cased Mainnetâ†’`"mainnet"`).
Same for `address --json`. Either pass `CliNetwork` through to the json
emit site, or add a small `cli_network_str(bitcoin::Network) -> &'static str`

### M3 â€” Third BIP 84 receive address not pinned

BIP 84 publishes only receive 0/1 and change/0 vectors; the existing
`tests/address_derivation.rs` has those three. SPEC's "receive 0..=2"

**Remediation:** drop the third receive index from the SPEC, or explicitly
direct the IMPLEMENTATION_PLAN to derive index 2 via rust-bitcoin in-test
(same approach as testnet). Cleanest: pin only 0/1/change-0 (BIP-published)
and use rust-bitcoin in-test for any additional indices.

### M4 â€” Wallet-policy gate has two distinct triggers; SPEC conflates them

The "address requires wallet-policy mode..." message is named in both
Â§Exit codes and Â§Implementation surface as the trigger for `--key`-omitted
template paths AND for phrase paths with no pubkeys. These are two distinct
checks: clap-level `requires` fires for the former, runtime
`is_wallet_policy()` fires for the latter.

**Remediation:** split the two error triggers in Â§Exit codes with distinct
messages. Suggested:
- clap-level: `"--key @i=<XPUB> required when --template is supplied"`
- runtime: `"address requires wallet-policy mode (Pubkeys TLV); supply --key @i=XPUB or use a wallet-policy-mode phrase"`
